# Обновления API (08.05.2026)

Этот документ описывает критически важные изменения в API, на которые необходимо обратить внимание фронтенд-разработчикам.

## 1. Новая модель истории транзакций (Transaction History)

Мы полностью переработали архитектуру и контракты для получения истории транзакций пользователя (`TransactionService.GetHistory`). Старая плоская структура заменена на группированную, так как в системе появились сложные многоуровневые финансовые операции (например, обмен валют или перевод, который вызывает покупку).

### Что изменилось?
- **Группировка (`Group`)**: Транзакции теперь приходят не по одной, а объединены в **Группы**. Группа — это единая бизнес-операция. Например, обмен USDT на WinCoins в рамках одной группы будет содержать две записи: списание USDT и зачисление WinCoins.
- **Вложенные записи (`Entry`)**: Каждая группа содержит список проводок (`entries`). Проводка имеет:
  - `currency_id` (валюта проводки)
  - `direction` (Debit — списание, Credit — зачисление)
  - `amount` (сумма, всегда положительная строка)
  - `details` (объект с деталями конкретной операции)
- **Нормализация данных**: Вместо того чтобы дублировать полные объекты пользователей и слотов внутри каждой транзакции, API теперь возвращает их отдельными массивами (`accounts`, `distributors`, `slots`) на уровне корневого ответа. 
  - *Для фронтенда*: Вы должны нормализовать эти списки (например, сложить в `Map` по `id`) и при рендере деталей транзакции связывать идентификаторы (например `sender_distributor_id`) с объектами из этих массивов.

### Структура `Details`
Поле `details` в каждой проводке — это `oneof` структура. В зависимости от типа операции, там будет разный payload. Примеры:
- `TransferIn / TransferOut`: Переводы между пользователями. Содержит идентификатор отправителя или получателя.
- `Exchange`: Обмен валют. Содержит идентификаторы торговой пары и заявки.
- `LicenseReferralBonus / TradeReferralBonus`: Реферальные бонусы. Указано, от кого пришел бонус и с какого уровня глубины.
- `MarketingVoucherPurchase`: Покупка ваучера.
- `ArenaPrize`: Выигрыш на арене. Содержит ID арены, номер цикла и занятое место.

### Инструмент для верстальщика: Заголовок `x-test-mock`
Чтобы вы могли протестировать и сверстать сложные сценарии (мультивалютные переводы, частичные покупки и прочее), мы добавили специальный режим. 
- **Как использовать**: Передайте в gRPC-запросе к `GetHistory` метаданные (заголовок) `x-test-mock: true` (или любое значение).
- **Результат**: Вместо реальной истории вы получите **полноценный заготовленный мокап** со всеми возможными краевыми случаями и видами транзакций.
- **Важно**: Мокап теперь "умный". Он берет вашего реального текущего пользователя (ваше фото, ник, distributor id) и подставляет в моковые данные, чтобы интерфейс выглядел естественно и можно было тестировать отображение собственных аватарок. В качестве "второй стороны" в переводах берется реальный случайный пользователь из базы.

## 2. Справочник Арен (Dictionary API)

Добавлен список доступных арен в публичный справочник системы.

- **Где**: Метод `DictionaryService.GetPublic`
- **Новое поле**: `repeated biconom.types.Arena arenas = 15;`
- **Для чего нужно**: Возвращает базовые метаданные всех арен (ID, строковое имя, общее количество завершённых и активных циклов). Это "легковесная" модель — она не содержит тяжелой информации вроде таблицы лидеров или конфигурации. Поле нужно для того, чтобы на старте приложения загрузить список арен и использовать их в фильтрах, навигации или истории транзакций.
