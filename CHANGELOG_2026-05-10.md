# Обновления API (10.05.2026)

Этот документ описывает изменения в API, касающиеся отображения истории транзакций пользователя.

## 1. Изменение структуры данных проводки (Entry) в истории транзакций

Мы упростили структуру возвращаемых данных для проводок в `TransactionService.GetHistory`. Ранее каждая проводка содержала отдельное поле `direction` (enum), указывающее направление движения средств (списание или зачисление), а также поле `amount` с положительным значением. 

Для упрощения логики на стороне фронтенда мы изменили этот механизм.

### Что изменилось?
- **Удалено поле `direction`**: `enum Direction` и поле `direction` были полностью удалены из сообщения `Entry`.
- **Знаковое поле `amount`**: Теперь направление движения средств зашито прямо в строковое поле `amount`. 
  - Если операция является списанием средств (Debit) и уменьшает баланс пользователя (например, вывод средств, покупка лицензии, комиссия), поле `amount` будет начинаться со знака минус (например, `"-50.00"`).
  - Если операция является зачислением средств (Credit) и увеличивает баланс пользователя (например, пополнение, бонус, подарок), поле `amount` вернётся просто как положительное число (например, `"50.00"`).

### Влияние на фронтенд
- Фронтенд-разработчикам больше не нужно проверять флаг `direction`, чтобы понять, окрашивать ли сумму в красный/зелёный цвет или добавлять минус визуально. Достаточно проверить наличие знака `-` в начале строки (или парсить как число).
- Поле `amount` можно напрямую использовать для отображения в UI, так как знак списания уже включён в ответ от API.

### Моковые данные (`x-test-mock`)
Режим моковых данных также был обновлен. Если передать заголовок `x-test-mock`, сервер вернет тестовые данные, где все списания уже корректно отформатированы со знаком минус.

## 2. Новый вид транзакции: Скользящий матчинг бонус (Split Matching Bonus)

В систему добавлен новый вид транзакции — `split_matching_bonus`. Этот бонус выплачивается вышестоящим спонсорам от прибыли нижестоящего партнера (например, при клейме дивидендного пула).
На данный момент этот бонус доступен **только в моковом режиме** для подготовки фронтенда (логика его начисления еще в разработке).

**Поля в payload `SplitMatchingBonusDetails`**:
- `initiator_distributor_id` (`uint32`): Идентификатор дистрибьютора, от которого пошло вознаграждение (кто заклеймил прибыль).
- `depth_level` (`uint32`): Уровень глубины, на котором находится этот дистрибьютор относительно получателя бонуса.
- `original_matching_amount` (`string`): Изначальный объем вознаграждения, который распределялся вверх по структуре (из которого получатель забрал свой процент).
