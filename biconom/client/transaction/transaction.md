# Client Transaction Service

> Модуль, предоставляющий агрегированную и удобную для UI историю транзакций пользователя.

## Контекст

Ядро (MLM Service) использует двойную запись (`TransactionGroup` -> `Transaction` -> `TransactionEntry`), где отражаются не только балансы пользователя, но и резервы, системные комиссии и транзитные счета. Выдача таких данных напрямую на фронтенд заставит клиента собирать "пазл" бизнес-логики и отфильтровывать лишнее.

Данный API выступает в роли Backend-for-Frontend (BFF), инкапсулируя всю бизнес-логику и отдавая только визуальные блоки истории баланса, которые уже готовы к рендерингу в UI.

## Основные сущности

*   **HistoryGroup** — Контейнер бизнес-операции. Содержит агрегированный статус, тип операции (`RECEIVE`, `PURCHASE` и т.д.) и таймстемп. Соответствует `TransactionGroup` ядра.
*   **HistoryEntry** — Вложенные изменения баланса для текущего пользователя (списание или зачисление). Соответствуют `TransactionEntry`, отфильтрованным по `account_id` пользователя.
*   **GroupMetadata** — Набор специфичных для `HistoryGroup.Kind` полей. Например: логин отправителя для перевода, название пакета или хэш транзакции в блокчейне.

## Механизм пагинации

Для обеспечения скорости $O(1)$ и мгновенной подгрузки истории используется **индекс по проводкам** (`core.mlm.cache.transaction_entry.index_account_ts`).

### Алгоритм выборки
1. Запрос начинает чтение из индекса `BTreeSet<(account_id, created_at, entry_id)>` в направлении `DESC` от курсора.
2. Извлекается ID проводки ➔ ID транзакции ➔ ID группы транзакций (`group_id`).
3. Так как одна группа может иметь несколько проводок (например: покупка пакета списывает USDT и начисляет Токены), `group_id` агрегируются и дедуплицируются.
4. Как только набирается `limit` уникальных групп, сканирование останавливается. Курсором для следующей страницы становится последняя считанная проводка (ее `created_at` и `entry_id`).
5. Для собранных `group_id` поднимаются полные графы транзакций, фильтруются проводки текущего пользователя и формируются структуры `HistoryGroup`.

## Безопасность и Access Control
* Сервис доступен только в сессии пользователя (`ParsedIdentifier::Session`).
* В `HistoryEntry` попадают только проводки, где `account_id` строго равен `account_id` из токена авторизации.

## Совместимость с UI
Типы из `HistoryGroup.Kind` должны строго сопоставляться с иконками и цветовой схемой фронтенда, а значения из `direction` и `amount` формируют цвет суммы (+ зеленый, - красный / серый).
