# Сервис: GeoService

## 1. Описание

**`GeoService`** — это клиентский сервис карты геометок дистрибьюторов. Он позволяет дистрибьютору **поставить или снять собственную метку** на карте (opt-in), получить **лёгкий список чужих меток** по географической зоне с фильтрами (баланс WinTime, иерархия), и отдельно **догрузить полные профили** (дистрибьютор + аккаунт + пользователь с аватаром) по списку идентификаторов.

Разделение на «лёгкий список» и «догрузку профилей» сделано ради масштаба: карта может содержать до ~1 млн точек, поэтому список отдаёт минимум данных (идентификатор + координаты + баланс), а тяжёлые профили подтягиваются отдельным запросом только для видимых или выбранных точек.

**Авторизация**: активная сессия + право `DISTRIBUTOR_MAP_VIEW`. Владелец во всех методах определяется по сессии; явный `distributor_id` не передаётся.

## 2. Ключевые концепции

### Метка (`Pin`)

Метка дистрибьютора хранится по принципу opt-in: пока дистрибьютор её не поставил, на карте его нет. Метка несёт `distributor_id`, координаты (широта/долгота в градусах), баланс WinTime владельца на момент ответа и время последнего обновления. Если метки нет (в `GetDistributorsInfo`), возвращается «пустая» точка с нулевыми координатами и без `updated_at` — фронт отличает такой случай по отсутствию `updated_at`.

### Выборка в `ListPins`

`ListPins` всегда ищет по прямоугольной зоне (bounding box) — все четыре границы `min_lat`/`max_lat`/`min_lon`/`max_lon` **обязательны**. Ведёт пространственный индекс зоны, затем применяются фильтр иерархии (`scope`) и фильтр баланса (`≥ min_win_time`). Показываются только точки с меткой внутри зоны.

Границы зажимаются к физическим пределам (широта ±90, долгота ±180) — чуть выходящие значения из-за округления карты не роняют запрос. Перепутанные `min`/`max` по оси нормализуются (меньшее — нижняя граница, большее — верхняя). Отсутствие любой границы или `NaN` → ошибка `GEO_INVALID_COORDINATES`.

### Область видимости (`ScopeOptions`)

Параметр `scope` определяет, чьи точки попадают в выборку (единообразно с `AnalyticsService`):

- **`PERSONAL`**: только сам смотрящий.
- **`TEAM`**: поддерево (структура) смотрящего. Имеет дополнительные настройки:
  - `team_depth_limit`: ограничивает глубину поддерева. `0` — команда не обходится (только сам, если включён `include_personal`); `1` — первая линия; и т.д. Если не указан — вся доступная глубина.
  - `include_personal`: включать ли самого смотрящего в основной список `pins` (в дополнение к `my_pin`).
- **`COMPANY_WIDE`**: вся компания (весь проект), без учёта иерархии.

Смотрящий никогда не дублируется: его личная метка всегда возвращается отдельным полем `my_pin`, а в основной список `pins` он попадает лишь при `include_personal = true`.

### Только с меткой, без лимита

`ListPins` всегда возвращает **только тех, у кого есть геометка** — те, кто её не поставил, на карте не показываются. Лимита на количество точек нет: отдаются все подходящие под зону/баланс/иерархию. Ответ лёгкий (только id + координаты + баланс), детали догружаются отдельно.

### Ленивая загрузка деталей (BFF)

`ListPins` намеренно возвращает лёгкий ответ (только точки). Полные профили владельцев запрашиваются отдельно через `GetDistributorsInfo` по списку идентификаторов — так фронт не грузит тяжёлые данные по всем точкам, а только по видимым или выбранным.

## 3. Описание методов (RPC)

### `rpc SetPin(biconom.types.Geo.Coordinates) returns (biconom.types.Geo.Pin)`
- **Назначение**: Поставить или обновить собственную геометку.
- **Параметры (`biconom.types.Geo.Coordinates`)**:
  - `latitude`: Широта (−90..90).
  - `longitude`: Долгота (−180..180).
- **Ответ**: Возвращает созданную/обновлённую метку `Pin` с текущим балансом WinTime владельца. Невалидные координаты → ошибка `GEO_INVALID_COORDINATES`.

### `rpc RemovePin(google.protobuf.Empty) returns (google.protobuf.Empty)`
- **Назначение**: Снять собственную геометку.
- **Использование**: Идемпотентно — если метки нет, вызов завершается успешно.
- **Запрос**: Принимает `google.protobuf.Empty` (параметров нет — владелец определяется по сессии).

### `rpc ListPins(ListPinsRequest) returns (PinsResponse)`
- **Назначение**: Получить лёгкий список геометок по прямоугольной зоне. Всегда возвращает только те точки, у кого есть метка; лимита на количество нет.
- **Параметры `ListPinsRequest`**:
  - `min_lat` / `max_lat` / `min_lon` / `max_lon`: Границы зоны — **обязательны** (все четыре). Зажимаются к ±90/±180; перепутанные `min`/`max` нормализуются; отсутствие границы или `NaN` → ошибка.
  - `min_win_time`: Минимальный общий баланс WinTime владельца (включительно).
  - `scope`: Область видимости по иерархии (см. `ScopeOptions`).
- **Ответ `PinsResponse`**:
  - `repeated pins`: Точки с меткой, прошедшие зону/баланс и иерархию (без смотрящего, если `include_personal` не задан). Только идентификатор + координаты + баланс — без профилей.
  - `optional my_pin`: Личная метка смотрящего (если есть), всегда, независимо от зоны и фильтров.

### `rpc GetDistributorsInfo(DistributorsInfoRequest) returns (DistributorsInfoResponse)`
- **Назначение**: Догрузить полные профили дистрибьюторов по списку идентификаторов (BFF).
- **Использование**: Вызывается после `ListPins` для видимых/выбранных точек, чтобы получить детали только по нужным дистрибьюторам.
- **Параметры `DistributorsInfoRequest`**:
  - `repeated distributor_ids`: Идентификаторы дистрибьюторов для догрузки. Дубликаты дедуплицируются, несуществующие id (вне `1..=последний_id`) отбрасываются. Число УНИКАЛЬНЫХ существующих id ограничено 1000 (`LIST_LIMIT_MAX`); сверх лимита → ошибка `LIST_LIMIT_MAX_EXCEEDED`.
- **Ответ `DistributorsInfoResponse`**:
  - `repeated pins`: Геометка (если есть) + баланс WinTime по каждому найденному идентификатору. Порядок — по возрастанию `distributor_id` (фронт сопоставляет по id, порядок не важен).
  - `repeated distributors`: Дедуплицированные профили дистрибьюторов.
  - `repeated accounts`: Дедуплицированные аккаунты. Аккаунт несёт вложенного пользователя (`User`) с аватаром — отдельного списка пользователей не требуется.

## 4. Примерный сценарий использования (Workflow)

**Задача**: Показать на карте дистрибьюторов с балансом WinTime от 10000 в текущем видимом прямоугольнике карты и подгрузить профили для видимых на экране.

**Решение**:

1. Клиент вызывает `ListPins`, передавая границы видимого прямоугольника карты и фильтры:
   ```
   {
     min_lat: 45.0, max_lat: 55.0,
     min_lon: 20.0, max_lon: 40.0,
     min_win_time: 10000,
     scope: { id: COMPANY_WIDE }
   }
   ```
   Сервер возвращает `PinsResponse` с точками в зоне (у кого есть метка и баланс ≥ 10000): лёгкий список (идентификатор + координаты + баланс) и личную метку `my_pin`.

2. Пользователь скроллит/зумит карту; фронт определяет видимые на экране точки и собирает их `distributor_id`.

3. Клиент вызывает `GetDistributorsInfo`, передавая идентификаторы только видимых точек:
   ```
   {
     distributor_ids: [ ...видимые id... ]
   }
   ```
   Сервер возвращает `DistributorsInfoResponse` с полными профилями (дистрибьютор + аккаунт с пользователем и аватаром) и точками (геометка + баланс). Фронт сопоставляет их локально по идентификатору.

## 5. Структуры данных

### `PinsResponse`
Лёгкий ответ `ListPins` (только точки).

- `repeated biconom.types.Geo.Pin pins`: Чужие точки, прошедшие фильтры (без смотрящего).
- `optional biconom.types.Geo.Pin my_pin`: Личная метка смотрящего (если есть).

### `DistributorsInfoResponse`
Ответ `GetDistributorsInfo` (полные профили по запрошенным id).

- `repeated biconom.types.Geo.Pin pins`: Геометка (опционально) + баланс WinTime по каждому найденному id (по возрастанию `distributor_id`).
- `repeated biconom.types.Distributor distributors`: Дедуплицированные профили дистрибьюторов.
- `repeated biconom.types.Account accounts`: Дедуплицированные аккаунты (с вложенным пользователем и аватаром).
