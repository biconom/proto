# CHANGELOG: 11-12 Мая 2026

За последние 24 часа был реализован масштабный пласт клиентского функционала, связанного с системой Web Push уведомлений, гибкой настройкой оповещений и улучшением email-рассылок.

Ниже приведено описание новых возможностей и инструкция по интеграции для Frontend-команды.

---

## 1. 🚀 Новые сервисы: Web Push и Управление Уведомлениями

Мы полностью внедрили систему доставки уведомлений, включая поддержку браузерных Web Push уведомлений и транзакционных Email-сообщений. 

### Управление подписками на Web Push
Добавлен сервис `biconom.client.push`. Он позволяет:
* Регистрировать новые устройства пользователя для получения Web Push уведомлений.
* Привязывать подписки к конкретной пользовательской сессии.
* Отвязывать устройства и удалять подписки при выходе из системы.

### Сервис Управления Настройками Уведомлений
У нас появился полноценный сервис управления уведомлениями — `biconom.client.notification_preferences`. 
Пользователи теперь имеют возможность гибко настраивать каналы получения оповещений:
* Включать или отключать **Email** и **Web Push** уведомления (в будущем и Telegram).
* **Подписываться на конкретные события (Топики)**: Теперь пользователь может сам выбирать, о чем он хочет получать уведомления (например, новые рефералы, начисления средств, обновления статуса заявок и т.д.), и для каждого события выбрать свой удобный канал связи.

### Подписка на события в реальном времени (Stream)
* В сервисе `biconom.client.confirmation` реализован метод `SubscribeToFieldStatus` (Stream RPC). Это дает возможность Frontend-приложению открывать постоянное соединение и моментально реагировать на изменение статуса подтверждения (например, когда пользователь подтвердил действие на другом устройстве или по ссылке из почты).

## 2. 🛡 Улучшения клиентского взаимодействия и Email

* **Персонализация писем:** Исправлен баг с пустым именем в письмах. Теперь система корректно извлекает логин пользователя при входе и подставляет его в приветствие (`Здравствуйте, UserName`).
* **Точное гео-определение:** Улучшена логика определения IP-адресов. Теперь в истории сессий и письмах безопасности всегда отображается реальный IP клиента.
* **CORS Политики:** Скорректированы настройки безопасности. Теперь API разрешает прямые кросс-доменные запросы с официальных доменов проекта (`https://win2.pro` и `https://biconom.com`).
* **Шаблоны писем:** Обновлены ссылки на службу поддержки (`@win2team`). Добавлено отображение inline-логотипов, благодаря чему письма загружаются быстрее и выглядят правильнее.

## 💻 Пример для Frontend: Как подключить Web Push и Настройки

Команде фронтенда для полноценной интеграции необходимо реализовать следующий флоу.

### 1. Получение публичного VAPID ключа
**Важно:** Не нужно хардкодить публичный ключ в `.env` фронтенда! 
Система разработана так, что ключ запрашивается с бэкенда динамически. Это обеспечивает безопасность и позволяет нам менять ключи без пересборки Frontend приложения.

Для этого используйте gRPC-метод `GetVapidPublicKey` в сервисе `biconom.client.push.PushService`. Метод доступен даже для Guest (не требует авторизации).

```javascript
// Запрашиваем ключ у бэкенда
const response = await PushServiceClient.GetVapidPublicKey({});
const vapidPublicKey = response.publicKey; // Строка в формате base64url
```

### 2. Регистрация Service Worker и создание подписки
```javascript
// 1. Регистрируем Service Worker
const registration = await navigator.serviceWorker.register('/sw.js');

// 2. Спрашиваем разрешение у пользователя
const permission = await Notification.requestPermission();
if (permission !== 'granted') throw new Error('Permission denied');

// 3. Подписываемся на пуши в браузере
// Вспомогательная функция для конвертации base64url в Uint8Array
const applicationServerKey = urlBase64ToUint8Array(vapidPublicKey);

const subscription = await registration.pushManager.subscribe({
    userVisibleOnly: true,
    applicationServerKey: applicationServerKey
});

// 4. Извлекаем ключи для отправки на бэкенд
const subJson = subscription.toJSON();
const endpoint = subJson.endpoint;
const p256dh = subJson.keys.p256dh;
const auth = subJson.keys.auth;
```

### 3. Регистрация устройства на Backend
Собранные данные нужно отправить через метод `Subscribe` в сервисе `biconom.client.push.PushService`. Это привяжет подписку к текущей сессии пользователя.

**Пример Payload (на основе gRPC proto):**
```json
{
  "device_id": "уникальный-идентификатор-устройства-или-сессии",
  "endpoint": "https://fcm.googleapis.com/fcm/send/...",
  "auth": "auth-ключ-из-подписки",
  "p256dh": "p256dh-ключ-из-подписки"
}
```

### 4. Интерфейс Управления Настройками (Preferences)
После того как устройство зарегистрировано, Frontend должен использовать сервис `biconom.client.notification_preferences`, чтобы отобразить пользователю панель настроек.

* Используйте метод для получения списка всех доступных событий (топиков).
* Предоставьте пользователю чекбоксы (Email, Push) для каждого события.
* Используйте метод `UpdatePreferences` для сохранения выбора пользователя:
```json
{
  "topic_id": "уникальный-ID-события",
  "email_enabled": true,
  "push_enabled": true,
  "telegram_enabled": false
}
```
